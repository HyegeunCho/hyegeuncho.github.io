유니티 에디터 크래시 트러블슈팅: UI 내비게이션과 레이아웃의 함정

유니티(Unity) 엔진으로 UI가 많은 게임을 개발하다 보면 예상치 못한 시점에 에디터가 멈추거나 종료되는 현상을 겪곤 합니다. 이번 포스팅에서는 게임 내 '도감 팝업'을 구현하던 중 발생한 에디터 크래시 현상과 그 원인을 분석하고 해결해 나가는 과정을 상세히 공유하고자 합니다.

[이 글을 작성하는 필요성]

일반적으로 UI 관련 크래시는 스크립트 오류나 무한 루프 등 코드 상의 문제로 접근하기 쉽습니다. 하지만 이번 사례는 코드의 논리적 오류가 아닌, 유니티 에디터의 시각화(Visualize) 기능과 과도한 UI 객체 수가 맞물려 발생한 특수한 케이스입니다. 이러한 문제는 디버그 로그조차 명확히 남지 않아 원인 파악에 막대한 시간을 소요하게 만듭니다. 따라서 본 사례를 기록으로 남김으로써, 비슷한 상황에서 레이아웃 연산이라는 잘못된 원인에 매몰되지 않고 빠르게 문제를 해결할 수 있도록 돕고자 합니다.

본론 (Main Content Description)

2.1. 문제 상황: Hierarchy 뷰에서의 원인 불명 크래시

도감 팝업 UI를 제작한 후, 데이터 바인딩과 레이아웃 구성을 점검하기 위해 에디터의 Hierarchy 뷰에서 게임 오브젝트들을 하나씩 선택하며 Inspector를 확인하고 있었습니다. 그런데 특정 시점부터 에디터가 갑자기 멈추더니 강제 종료(Crash)되는 현상이 발생했습니다.

특이한 점은 Game 뷰(인게임 플레이)에서 팝업을 조작할 때는 아무런 문제가 없었다는 것입니다. 오직 에디터 상태에서 Hierarchy의 오브젝트를 클릭하여 Inspector를 갱신할 때만 크래시가 발생했습니다.

2.2. 초기 가설과 실패: 레이아웃 연산의 폭주?

처음에는 제미나이(Gemini)와 함께 원인을 분석하며, UI 구성의 복잡성을 의심했습니다. 당시 도감 팝업은 스크롤 뷰 내부에 컨테이너를 중첩해서 사용하고 있었는데, 구조가 다음과 같이 상당히 무거웠습니다.

  • ∟ VerticalLayoutGroup
  • ∟ ContentSizeFitter
  • ∟ VerticalLayoutGroup
  • ∟ GridLayoutGroup (아이템 리스트)

이러한 중첩 구조 때문에 Inspector를 갱신할 때마다 캔버스 리빌드(Rebuild)와 레이아웃 연산 폭주가 일어나 에디터가 버티지 못하는 것이라 판단했습니다. 이를 해결하기 위해 캔버스 분리 및 연산 제한 등의 최적화를 시도했으나 크래시는 여전히 발생했습니다. 만약 레이아웃 연산 부하가 진짜 원인이었다면 인게임에서도 문제가 있어야 했으나, 인게임은 너무나 멀쩡했습니다.

2.3. 진짜 원인 발견: 버튼(Button)과 Navigation Visualize

문제 재현을 위해 테스트를 반복하던 중 결정적인 단서를 발견했습니다. 일반 이미지나 텍스트 오브젝트를 선택할 때는 문제가 없는데, 유독 '버튼(Button) 오브젝트'를 선택할 때만 크래시가 발생한다는 점이었습니다.

원인은 바로 유니티 에디터의 Navigation Visualize 기능이었습니다.

유니티 UI 버튼은 생성 시 기본적으로 Navigation 속성이 'Automatic' 등으로 설정됩니다.
에디터는 버튼이 선택되면 씬(Scene) 뷰에 버튼 간의 이동 경로(화살표)를 시각적으로 보여주는 기능을 수행합니다.
도감 팝업에는 수많은 아이템 버튼이 존재했고, 이 모든 버튼에 불필요한 Navigation 설정이 켜져 있었습니다.
결과적으로 에디터가 수많은 버튼의 네비게이션 경로를 한 번에 계산하고 그리려다 과부하가 걸려 사망한 것이었습니다.

2.4. 문제 해결: Navigation 제거 및 오브젝트 풀링 도입

현재 개발 중인 게임은 키보드나 컨트롤러를 지원하지 않는 마우스/터치 기반 게임이므로, 네비게이션 기능이 불필요했습니다.

먼저 모든 버튼의 Navigation 속성을 None으로 변경하여 시각화 연산 부하를 제거했습니다. 또한, 프로토타입이라 미뤄두었던 오브젝트 풀링(Object Pooling) 기능을 구현하여 씬에 존재하는 활성 오브젝트의 절대적인 수를 줄였습니다. 이 두 가지 조치 후, 에디터 크래시 현상은 완전히 사라졌습니다.

결론 (Conclusion)

이번 트러블슈팅을 통해 얻은 핵심 교훈은 두 가지입니다. 첫째, 이슈 발생 시 단순히 '무거워서'라고 단정 짓지 말고, '어떤 동작을 할 때(버튼 클릭 시)' 발생하는지 디테일한 조건을 찾아야 한다는 점입니다. 둘째, 오브젝트 풀링과 같은 필수적인 최적화 기법은 프로토타입 단계에서도 배제하지 말고 기본적으로 적용해야 한다는 것입니다.

향후 활용 계획

01. UI 프리팹 표준화

버튼이나 상호작용 가능한 UI 컴포넌트 생성 시, 키보드/패드 지원이 없는 프로젝트라면 기본적으로 Navigation 속성을 'None'으로 설정해두는 프리셋을 배포할 것입니다.

02. 기본 모듈 라이브러리 구축

오브젝트 풀링이 적용된 스크롤 뷰 모듈을 라이브러리화하여, 프로토타입 단계부터 성능 이슈를 원천 차단하는 데 활용할 것입니다.
